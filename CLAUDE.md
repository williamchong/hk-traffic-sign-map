# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

`pnpm` is broken in this environment — always use `corepack pnpm`.

- `corepack pnpm dev` — dev server (http://localhost:3000)
- `corepack pnpm lint` / `corepack pnpm typecheck` — must both pass before any commit
- `corepack pnpm generate` — static build to `.output/public`; also used as the final pre-commit gate (it exercises the SSR/prerender path the dev server doesn't)
- `corepack pnpm data:build` — fetch TD GML + derive face bearings + (re)build `public/data/traffic-signs.pmtiles`
- `corepack pnpm data:catalogue` — re-extract sign pictograms + `app/data/signCatalogue.json` from the Index Plan PDFs (Sonnet 4.6 vision; needs `ANTHROPIC_API_KEY`, costs ~$0.15 to rebuild all 16 sheets)

There is no test suite; verification is lint + typecheck + `generate`, plus a real-browser smoke test via the chrome-devtools MCP when map behaviour changes (the map only runs client-side with WebGL).

Dev gotcha: after dependency/cache changes the Vite dep-optimizer can 504 ("Outdated Optimize Dep") and the map never mounts. Fix: kill dev, `rm -rf node_modules/.vite node_modules/.cache .nuxt/cache`, restart.

## Two runtimes — do not cross-import

`scripts/*.mjs` are Node build-time tools; `app/` ships to the browser. Shared constants (tile layer name, category/`SIGNID` conventions) are **intentionally duplicated** across the boundary rather than imported. `app/composables/useSignCategories.ts` keys must stay consistent with the category logic in `useSignCatalogue.ts`; `scripts/sign-layers.mjs` is the manifest the pipeline tags features with.

## Data pipeline (build-time)

1. `fetch-data.mjs` → raw GML into `data/raw/` (gitignored, resumable). Pulls the tiled `SIGN_LAYERS` plus the build-time-only `BUILD_TIME_DATA` (currently just `DTAD_RD_MARK_LINE` — ~155 MB, used to derive face bearings; never tiled).
2. `compute-bearings.mjs` → spatial-joins each `DTAD_TS_ABV_PT` feature to the nearest road-marking segment ≥3 m, takes the segment's compass bearing as the road tangent, and flips by 180° on the kerb-opposite side (cross-product of tangent × (sign − projected point)) so opposite-carriageway signs come out 180° apart. Output: `data/raw/_face_bearings.json` keyed by `FEATUREID` → degrees. Covers ~76 % of the 178k signs (off-network signs and gantries miss); the rest fall through to upright at runtime via `coalesce`. The remaining ~22 % of nearby same-code pairs that come out at mid-angles are mostly junction/slip-road edge cases where the nearest marking isn't aligned with the actual carriageway the sign serves — accepted tradeoff vs the alternative of no rotation at all.
3. `build-tiles.mjs` → `ogr2ogr` reprojects each layer **EPSG:2326 (HK1980 Grid) → EPSG:4326**, injects a `category` property per the `sign-layers.mjs` manifest *and* (for the `traffic-sign-abbreviation` layer only) the `FACE_BEARING` looked up by `FEATUREID` from step 2, then `tippecanoe` packs the result into **two** PMTiles archives from the same combined input (source-layer `signs` in both): `public/data/traffic-signs.pmtiles` (~19 MB, **thinned** via `--drop-densest-as-needed` — the LOD overview for the unfiltered/category view) and `public/data/traffic-signs-full.pmtiles` (~41 MB, **retain-all** via `-r1 --no-feature-limit`, restricted to the abbreviation class with `-j` — the sign-ID-filter distribution view). One pyramid can't serve both because tippecanoe's feature-dropping is **filter-blind** (decided per-tile before the runtime filter is known): a thinned pyramid gives a clean overview but a filtered subset of it shows only drop-lottery survivors at low zoom (the bug behind commit `5a1ad4e`), while a retain-all pyramid fixes the filter but mushes the overview. The runtime swaps which source the sign layers read by `filterMode` (see Runtime architecture). One combined `?v=<hash>` (over both files' bytes) busts the byte-range cache. tippecanoe maxzoom settles at 15 (MapLibre overzooms past it — fine for points).
3. `build-sign-catalogue.mjs` (independent) reads each rendered TD **Index Plan** page (a table of `[No. | Symbol | Description]` column-groups) with Sonnet 4.6 vision and writes `public/signs/<CODE>.png` + `app/data/signCatalogue.json` (`{ tier, group, desc? }`). Three safety nets enforce the cardinal "a missed sign degrades to a dot — a *mislabelled* sign must never ship" rule:

   - **No.-column grid bind** (the row anchor): two VLM calls per page — a full-page read for descriptions + group count, then a read of a montage of every `(group, rowBand)` **No.-cell** cropped in reading order (`readNoGridVLM`). The montage reply MUST be exactly R×C; a misshapen one means the bind would be misaligned, so it is re-asked a few times and then the sheet is **skipped** (never force-fit). Because each code is read from the same pixels its symbol is cropped from, the bind can't drift across blank/"SYMBOL NOT AVAILABLE" rows — the failure an earlier `y`-hint approach had (got a recovery reverted; see `git log` / memory). The symbol-column x is a fixed **606 px lattice** (origins ~472/~776); the No.-cell is located from vertical rules anchored on the symbol cell's **left edge** (centre-anchoring grabs spurious in-pictogram rules).
   - **GML SIGNID filter**: a model-emitted code is only kept if it appears at least once in `data/raw/DTAD_TS_ABV_PT.gml` (the ground-truth set of codes on real road signs in HK). Codes the app would never reference are dropped — loss-less filter that also catches the model's occasional sequential-number hallucinations.
   - **Human gate** (`--propose` / `--commit`): recovery stages crops + a `verify.png` (each candidate's SOURCE `[printed No. | pictogram]` strip, labelled with the bound code) + a `manifest.json` under `/tmp/sign-recovery`, writing NOTHING to the repo. A human confirms every printed-number↔label before `--commit [--reject TS#,TS#]` promotes the approved crops. **Known limitation:** sheets with per-group rowspans (informatory: 601-700, 2601-2717, 3601-3705, 3811-3936) or a degenerate symbol-band fit can't bind via the shared R×C grid — they skip or get rejected at the gate rather than mislabel.

   Modes: bare `data:catalogue` does a full write-through rebuild merging into the existing catalogue; `--wipe` clears `public/signs/` + the catalogue first. `--propose`/`--commit` is the gated recovery path above. `--sheet "<pattern>"` processes only matching sheets. PASS 1 descriptions are **additive-only** (fill a missing `desc`, never clobber an existing one — re-running used to corrupt good descriptions when the VLM mis-paired a neighbour's text); `--update-desc` re-enables overwrite. Authoritative bilingual meanings still live in hand-curated `app/data/signDescriptions.json` (`{ "<CODE>": { en?, zh? } }`, sourced from the TD Road Users' Code) which the runtime prefers over `desc` and is edited **independently — no pipeline re-run**. The Road Users' Code figure numbers do **not** map to `SIGNID`, so that file is curated by hand, never auto-derived. Never equate Cap 374G legal figure numbers with TD `SIGNID` above the low regulatory range.

   History (git log): an earlier tesseract + 3-layer-LIS pipeline (`b00a1df`…`f111fd8`) extracted ~457 codes before being replaced by the VLM extractor (`56a1a84` onwards) at 753+ codes, ~$0.15 per full rebuild vs minutes of OCR. The LIS scaffolding became archaeology when the reader's error model changed.

Key dataset fact: **`SIGNID` exists only on the `traffic-sign-abbreviation` class.** Pole classes (`DTAD_TS/PS/DS_POLE_PT`) have no `SIGNID` and no sign content — they are bare posts. Only abbreviation features can resolve to a pictogram; everything that maps to category key `none` (poles) is never rendered.

## Runtime architecture (the part that needs multiple files)

- `useSignCatalogue.ts` is the hub: loads `signCatalogue.json` + `signDescriptions.json`, exposes **`signDescription(signId, locale)`** (curated zh/en override → curated en → OCR `desc` → null; English shows verbatim in the zh UI when no zh override exists), derives `codesByTier`/`codesByGroup`, and exports **`categoryKeyExpr`** — one static MapLibre expression mapping any feature to its category key: a catalogued Index-Plan group (`regulatory|warning|informatory|supplementary|temporary`), else by tile `category` → `tourist`/`other-traffic`, else `none`. `TIER_LOD` defines per-tier `minzoom` and the top-of-ramp `icon-size`; `SIGN_FIRST_SIZE` is the shared normalised height every tier shows at its reveal zoom.
- `useSignCategories.ts` = legend/filter rows + colours, keyed by those category keys.
- `useTrafficLayers.ts` = singleton state: `enabled` per category, `selectedSign`, and `mapFilter` = `['in', categoryKeyExpr, enabledKeys]` — the one filter every layer rides.
- `TrafficMap.vue` builds the map: an always-on `sign-points` circle (coloured by `categoryColor` = `match` on `categoryKeyExpr`) as the baseline marker; per-tier `sign-tier-N` symbol layers (`minzoom` = tier minzoom, `icon-size` ramps from shared `SIGN_FIRST_SIZE` up to the tier size, collision fully disabled); a `sel` GeoJSON highlight overlay (halo+dot+icon) moved to the top on every selection; click collects all signs in a 6px box, de-dupes, and cycles through them on repeat clicks. **Two vector sources** are registered (`signs-lod` = thinned overview archive, `signs-full` = retain-all archive); the sign layers read the one `sourceForMode(filterMode)` picks — `signs-lod` in category mode, `signs-full` in sign-ID mode. MapLibre can't repoint a live layer's source, so a `filterMode` flip calls `removeSignLayers()`/`addSignLayers(source)`; the sign layers are inserted **beneath `sel-halo`** (the `beforeId`) so the selection overlay stays on top across the swap (this replaced the old "moveLayer above async-added tiers" need — that moveLayer is now a defensive re-assert).

### Invariants that caused churn — keep them

- **Collision is disabled** (`icon-allow-overlap`/`icon-ignore-placement` always `true`) because every point is a real installed sign with a 1:1 ground meaning, so MapLibre must never drop or nudge one for a neighbour. The old "constant `icon-size`, never zoom-interpolated" rule existed *only* to stop growing icons from collision-hiding already-shown signs — with collision off that coupling is gone, so it was deliberately retired. Do not reinstate per-zoom collision.
- **Sign icons rotate by `FACE_BEARING`, not the raw TD `ANGLE`.** TD's `ANGLE` is the MicroStation symbol-cell rotation ("same as Ustn angle" per the dataspec) and 59 % of same-code signs <30 m apart share an `ANGLE` — opposite carriageways look identical when fed to `icon-rotate` (commit `fde0258` proved this, `42c343a` reverted it). `FACE_BEARING` is the build-time-derived compass bearing from `compute-bearings.mjs` (see Data Pipeline step 2) — ~76 % coverage; the missing ~24 % `coalesce` to `0` and render upright. `DTAD_RD_AL_LINE` looked tempting but covers only ~178 km of HK's ~2,200 km road network — `DTAD_RD_MARK_LINE` (lane markings, the densest TAD road-geometry layer) was the winning input.
- `icon-opacity` is **zoom-faded** (0.55 at z13 → 0.9 at z17, held at 0.9 — never fully opaque) on the tier layers. With collision off, signs pile up zoomed out and can still overlap at max zoom; the hard fade lets the stack show through while crowded, and the permanent 0.9 ceiling keeps residual high-zoom overlaps legible-through. Per-*sign* "is it overlapped" opacity is **not expressible in MapLibre** (collision can only hide, not report) — don't attempt it; this fade-by-zoom plus the constant ceiling is the deliberate substitute.
- `icon-size` now ramps per tier: every tier starts at the **shared** `SIGN_FIRST_SIZE` at its reveal `minzoom` (normalised on-screen height — pictograms are all 120px tall) and interpolates up to the tier's `size` at `MAX_ZOOM`. Keep the *first-display* size shared across tiers (visual consistency); only the top-of-ramp size is tier-specific (legibility for complex signs).
- The always-on dot is the fallback so a sign is **never invisible** below its tier's minzoom. This was tried both ways; the always-on dot won because the alternative left a confusing zoom band with neither dot nor sign.
- All instances of a `SIGNID` are in exactly one tier ⇒ one appearance zoom.

## Deployment

Static (`corepack pnpm generate` → `.output/public`). **The host MUST serve HTTP `Range` (`206 Partial Content`)** — PMTiles reads both archives (~19 MB overview + ~41 MB full) in byte-range slices, so a given view only pulls the slices it needs. GitHub Pages, Cloudflare Pages, Netlify, S3+CloudFront, nginx all do. `nuxt preview` does **not** (returns the whole file with `200`) — local checks only.

Commit style: gitmoji, phase-tagged (see `git log`).
